# PLAN — 남은 작업

## 🔴 보안 (미착수, 우선순위 높음)

운영 시작 전에 최소 1·2번은 처리한다.

### 1. JWT Secret 교체 확인
- 개발 중 `SUPABASE_JWT_SECRET` 값이 외부(대화)에 노출된 적 있음.
- 이 값이 유출되면 **RLS를 통째로 우회**할 수 있다 (임의 회원 사칭 토큰, `service_role` 권한 토큰 위조 가능).
- 조치: Supabase → Settings → JWT Keys → 재생성 → Vercel env 갱신 → Redeploy.
- **상태: 확인 필요**

### 2. 로그인 무차별 대입 방어
- 현재 회원 로그인 = `이름 + 전화번호 뒤 4자리`. 경우의 수 1만, **시도 횟수 제한 없음**.
- 피해 범위는 RLS 덕에 제한적이다: 뚫려도 **본인 예약만** 조회/취소 가능하고
  타인 명단·전화번호·서명은 노출되지 않는다. 다만 **남의 예약을 취소하거나
  주간 횟수를 소진시키는 장난**은 가능하다.
- 선택지 (택1):
  - **(a) 시도 제한**: 이름당 실패 5회 → 10분 잠금. 테이블 하나 + 카운터. 회원 경험 변화 없음.
  - **(b) PIN 추가**: 전화 뒤 4자리 + 회원 PIN 4자리. 더 안전하지만 회원이 하나 더 외워야 함.
- **상태: 방식 미정**

### 3. 관리자 계정 보호
- 관리자가 뚫리면 전 회원 이름·전화번호·서명이 노출된다.
- 강한 비밀번호 필수. 공용 PC 로그인 지양.
- **상태: 운영 수칙으로 관리**

### 4. 백업
- Supabase Free는 자동 백업이 제한적이다. 정기적으로 회원/이용권 테이블을 내보내 둘 것.
- **상태: 미착수**
